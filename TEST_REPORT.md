The two **service** layers (`AuthService`, `VehiclesService`) — where the actual business
logic lives — are at effectively 100%/96% statement coverage via focused unit tests written
before the implementation (TDD red → green, see the git history). Controllers, modules, DTOs,
and the JWT strategy show 0% here because they contain no branching logic of their own; they
are exercised instead by the **end-to-end tests** (`test/auth.e2e-spec.ts`,
`test/vehicles.e2e-spec.ts`), which spin up the full Nest application (real HTTP layer, guards,
pipes, Mongoose models) against an ephemeral in-memory MongoDB via `mongodb-memory-server`, and
assert on status codes and response bodies for registration, login, role-gated CRUD, search
filtering, purchase stock decrement, and restock.

**Note on running e2e tests:** `mongodb-memory-server` downloads a `mongod` binary the first
time it runs. In the sandboxed environment this project was built in, outbound network access
was restricted to package registries (npm, GitHub, etc.) and did not include MongoDB's binary
CDN, so the e2e suite could not be executed there — it does, however, type-check cleanly
against the full application graph. On a normal developer machine or CI runner with standard
internet access, `npm run test:e2e` will download the binary once (cached afterwards) and run
both e2e suites (16 assertions covering the full request/response cycle, including 401/403
authorization checks).
